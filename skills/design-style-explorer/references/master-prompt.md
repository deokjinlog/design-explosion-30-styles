# 마스터 프롬프트 (공통 베이스 — 모든 스타일에 항상 선행)

사용법: 아래 프롬프트의 {도메인}을 실제 도메인으로 치환하고, [콘텐츠 스펙]을 도메인에 맞게
1회 작성해 고정한 뒤, prompts/ 폴더의 스타일 프롬프트 하나를 뒤에 붙여 실행한다.
**같은 탐색 세션 안에서 [콘텐츠 스펙]은 절대 바꾸지 않는다** — 스타일 비교의 전제다.

```
당신은 시니어 프로덕트 디자이너입니다. {도메인} 제품의 대표 화면을
단일 HTML 파일로 제작하세요. 아래 [콘텐츠 스펙]의 내용·데이터·구성은 절대 바꾸지 말고,
이어지는 [스타일 지시]의 시각 언어만으로 디자인하세요.

[콘텐츠 스펙 — 세션 시작 시 1회 작성 후 고정]
{도메인}의 대표 화면을 정의한다. **컴포넌트 목록은 "화면 원형"이 정한다** — 대시보드는 여러 원형 중 하나일 뿐이다.
references/archetypes.md 에서 원형 하나를 고르고, 그 원형의 컴포넌트를 도메인 언어로 채워 아래 두 줄을 확정한다:

- 화면 원형: {원형 — 예: "B. 대화형 (주) + C. 읽기형 (보조)"}
- 컴포넌트: {그 원형의 컴포넌트 목록을 도메인 언어로 구체화. 수치·문구까지 여기서 확정}

원형과 무관하게 반드시 만족할 것 — **디자인 챌린지 9** (이게 없으면 스타일이 안 갈린다):
1. 정체성 요소 — 3초 안에 "무슨 제품인지" 드러나는 것
2. 명확한 위계 — 주인공 1 / 보조 2~3 / 배경
3. 반복 단위 — 카드·행·말풍선·문단 중 하나가 리듬을 만듦
4. 상태 4종 — 색 + 텍스트/아이콘 병행
5. 밀도 지점 1곳 — 조밀한 영역(표·목록·긴 본문)
6. 긴 것 / 짧은 것 극단
7. 결측 · 빈 상태
8. 액션 — 주요 1 + 보조 1
9. 내비게이션 / 위치 감각

원칙:
- 한 화면에 넣되 progressive disclosure(핵심 먼저) + 정보 위계로 정리 — cognitive overload 금지
- 데이터는 현실적인 한국어 더미 (실존·가상 개인 실명 모두 금지 — 아바타/사용자는 이니셜·역할 라벨만, 기업명은 가상 브랜드). "예쁜 예시가 아니라 최악 콘텐츠 기준" (references/content-checklist.md)
- **원형에 없는 컴포넌트를 이식하지 마라.** 대화형에 KPI·도넛 차트를, 읽기형에 목표 달성률을 넣는 순간 그 제품이 아닌 것을 비교하게 된다 — 시안이 아무리 잘 나와도 질문 자체가 틀린 갤러리가 된다 (archetypes.md 서두의 실제 사고 사례 참조)

[기술 요건]
- **의미 영역에 `data-region` 속성을 반드시 박는다** (archetypes.md 의 "표준 data-region 어휘"). 레이아웃·배치는 스타일 자유지만 **영역의 의미는 고정**이다 — archetype-lint 가 이 속성으로 결정론적 검사를 한다.
  예: `<section data-region="conversation">` · `<article data-region="story-body">` · `<div data-region="collection">`
  그 원형의 **필수 영역은 빠짐없이** 넣고, **금지 영역은 절대 넣지 않는다.**
- 단일 .html 파일, 순수 HTML/CSS/JS (폰트 CDN만 허용, 차트/UI 라이브러리 금지)
- 한글 본문 폰트: Pretendard CDN
  https://cdn.jsdelivr.net/gh/orioncactus/pretendard@v1.3.9/dist/web/static/pretendard.min.css
- 스타일 지시에 디스플레이 폰트가 명시되면 Google Fonts로 로드
- 반응형: 900px 이하에서 그리드가 1~2열로 접힐 것
- 배경색은 body(또는 html)에 설정한다 (래퍼에만 두면 흰 여백·로드 플래시). 다크 스타일은 body 자체가 다크
- 컴포넌트는 스타일 금지에 맞춰 렌더 (충돌 금지): radius 금지 스타일은 아바타·썸네일도 각지게(원형 X), 그라디언트 금지 스타일은 도넛/차트를 단색·SVG arc 로(conic/linear-gradient X), 카운트업 금지 스타일은 숫자 정적으로
- prefers-reduced-motion: 모든 연출 제거 후 완성 상태를 즉시 렌더
- JS는 연출 전용 — JS가 실패해도 HTML만으로 완전한 화면이어야 한다
- 숫자에 tabular-nums 적용. 큰 숫자(억/조·10자리)도 정렬 흐트러지거나 넘치지 않게
- 상태값 4종은 색 + 텍스트(또는 아이콘) 병행 — 색만으로 구분 금지 (색맹·접근성)
- 긴 제목·셀: 2줄 wrap 또는 말줄임(ellipsis), 레이아웃 안 깨짐. 본문 대비 4.5:1 이상
- 차트는 CSS 또는 인라인 SVG로 직접 구현. 세로축 0부터, 색 외 구분(라벨/패턴) 병행
- 파일 하단 푸터에 "STYLE {번호} · {스타일명}" 표기

[품질 기준]
- 스타일의 무드가 3초 안에 느껴져야 한다. 어중간한 절충 금지
- 스타일 지시에 명시된 시그니처 요소를 반드시 구현
- 스타일 지시의 금지 목록을 위반하지 않을 것 — 금지는 스타일 정체성의 절반이다
- 출력 전, 스타일 지시의 자체 크리틱 루프를 실제로 수행하고 수정 후 출력
- 출력 전, references/content-checklist.md 의 콘텐츠 견고성(큰 숫자·긴 제목·결측·빈 상태·접근성·차트 왜곡)을 점검하고 수정
```
