개발방향: copy-explosion (카피 폭발)

이 페이지는 copy-explosion-tech-design.md 의 사람이 읽기 편한 시각화 사본입니다 (내용 1:1 동일).

다음 단계 안내: 이 문서는 기술 설계서입니다 (아키텍처 / 컴포넌트 / 데이터 / 인터페이스 / 결정 / 위험 / 테스트 전략). copy-explosion-requirements.md (PRD) 를 기반으로 작성되고, 다음 단계 copy-explosion-implementation-plan.md (단계별 계획) 의 입력이 됩니다. 다음 단계로 writing-plans skill 을 호출해서 구현 계획을 만드세요. 단계별 구현 task 는 여기 박지 마세요.

1아키텍처 개요

copy-explosion 은 독립 플러그인 copy-explosion-30-angles 안의 스킬 1개(copy-angle-explorer) 다. design-explosion-30-styles 와 완전 대칭 구조를 갖되, 변수축이 "스타일" 이 아니라 "메시지 각도" 다.

핵심 원칙: 제품 사실은 고정, 표현 각도만 변수. design-explosion 의 "콘텐츠 고정, 스타일 변수" 를 그대로 옮긴다. 이 원칙이 아키텍처의 두 지점에 물리적으로 박힌다 — ① 모든 시안이 하나의 고정된 제품 사실 스펙을 공유하고, ② 모든 시안이 하나의 중립 히어로 템플릿을 공유한다. 그래서 갤러리에서 눈에 보이는 차이는 카피뿐이다.

3데이터 모델/스키마 변경

DB 없음 — 파일/객체 스키마만 정의한다 (본 스킬은 순수 파일 기반, design-explosion 과 동일).

3.1 제품 사실 스펙 (30 시안 공유·불변 — copy-lint 의 진실성 기준)

제품명        : 문자열
한줄설명      : 문자열
타겟          : 문자열 (누구를 위한 것)
핵심이득[]    : 문자열 배열 (1개 이상)
근거·사실[]   : 문자열 배열 — 수치·고유명사·수상·고객사 등 "말해도 되는 사실"의 전체 집합
언어          : 기본 "ko"
톤힌트?       : 선택 (예: "전문적", "친근")

근거·사실[] 이 copy-lint 의 화이트리스트다. 여기 없는 정량 표현(숫자·%·통화·고유명사·"1위/최초/유일" 류)이 시안에 등장하면 지어낸 것으로 본다 (§5 결정2).

3.2 각도 프롬프트 파일 스키마 (angle-NN-*.md)

design-explosion 의 style-NN-*.md 미러: 각도명 · 패밀리 · 설득 구조 · 헤드라인 공식 · 톤 · 좋은 예/나쁜 예 · 금지 · 크리틱 루프.

3.3 시안 산출물

  • angle-NN-<slug>.html — 중립 히어로 템플릿에 카피만 채운 자기완결 HTML. 푸터 ANGLE {n} · {각도명} · {패밀리} (FR-9).
  • gallery.html — 패밀리(A~F)별 그룹 카드 그리드 + 미리보기 + 새 탭 링크.

3.4 30 각도 × 6 패밀리 (확정 — angle-taxonomy.md 근거)

5핵심 결정 + 대안 비교

결정 1 — 별도 플러그인 repo (copy-explosion-30-angles)

대안

(b) design-explosion 플러그인에 2번째 스킬 추가 / (c) 하위 폴더에만.

선택 이유

마켓플레이스 대칭(짝 스킬로 나란히 노출)·독립 설치·관심사 분리. design-explosion 릴리즈 주기와 얽히지 않는다.

트레이드오프

갤러리 생성기·퍼널 프레임워크가 양쪽에 중복. → 각 스킬이 자기완결이라 실질 공유 코드가 적고(각각 HTML 골격이 다름), 복붙 각색 비용 < 결합 비용. 수용.

결정 2 — 하이브리드 카피-린트 (이번 설계의 심장)

대안

(b) 기계적만 + 나머지 [MANUAL] / (c) 각 시안 LLM 판정 서브에이전트 풀.

선택 이유

진실성 위반 중 가장 위험하고 가장 흔한 형태가 정량 클레임(없는 숫자·고객사·수상)이고 이건 정규식+화이트리스트로 결정론적으로 잡힌다. 나머지 미묘한 진실성·각도 구별성만 판정 에이전트로 넘기면, (b)보다 자동성이 높고 (c)보다 가볍다.

구조

copy-lint.py 구조 (design-explosion style-lint.py 미러):

  • FAIL(기계적): 헤드라인/서브 길이 초과, CTA ≠ 1개, 히어로 1뷰포트 위반 신호.
  • FAIL(정량 클레임): 시안에서 숫자·%·통화·"1위/최초/유일/1등" 류·대문자 고유명사를 추출 → §3.1 근거·사실[] 화이트리스트에 없으면 위반.
  • WARN: 휴리스틱 특성상 오탐 가능 항목(고유명사 오탐 등)은 WARN 으로 낮춰 사람이 확인.
  • [MANUAL]/판정 에이전트: 미묘한 진실성(과장·함의)·각도 구별성(두 시안이 사실상 같은 각도).
트레이드오프

정량 추출 휴리스틱의 오탐/미탐. → 위험한 쪽(미탐=지어냄 통과)은 판정 에이전트가 2차로 거르고, 오탐은 WARN 등급으로 흡수.

결정 3 — 30 파일 + master-copy-prompt (1 taxonomy 파일 아님)

대안

30 각도를 1개 taxonomy 파일에 정의.

선택 이유

병렬 오염 방지 — 각 에이전트가 자기 angle 파일만 읽어야 각도가 안 섞인다(1파일이면 에이전트가 30개를 다 읽어 톤이 평균화됨). design-explosion 과 대칭.

트레이드오프

파일 30개 관리 부담. → angle-taxonomy.md 인덱스 표로 상쇄.

결정 4 — 중립 단일 히어로 템플릿 (design-explosion 스타일 재사용 아님)

대안

design-explosion 30 스타일 중 하나를 재사용.

선택 이유

FR-8 "메시지만 변수". 스타일이 섞이면 비교가 오염된다. 스타일 결합은 확정 후 design-explosion 핸드오프의 몫(§4).

결정 5 — 히어로 길이 상한 수치 (한국어 기준)

헤드라인 ≤ 60자, 서브카피 ≤ 140자, CTA 정확히 1개, 히어로 1뷰포트 완결.

대안

상한 없음(자유). → 각도 간 비교가 길이차로 오염되고 "히어로" 형식이 무너져 기각. 수치는 구현 중 조정 가능(설정값).

7테스트 전략

  • 단위 (copy-lint.py) — design-explosion style-lint.py 에 테스트가 없는 점을 개선해 이쪽은 픽스처 테스트를 둔다:
    • 히어로 제약: 길이 초과/CTA 2개 픽스처 → FAIL, 정상 → PASS.
    • 정량 클레임: 제품 스펙에 없는 "1만 고객"·"업계 1위"·"삼성전자" 삽입 픽스처 → FAIL, 스펙 내 사실만 → PASS.
  • 통합(스모크) — 목 제품 스펙 1개로 파이프라인: angle 파일 30개 존재, 각 시안 푸터 메타 형식, 갤러리가 6패밀리로 그룹.
  • 커버리지 우선순위: FR-11~14(카피-린트) 집중 → 이 제품의 최상위 규율(NFR-1)이 여기 걸려 있다. 그다음 FR-4(30각도/6패밀리 존재).
