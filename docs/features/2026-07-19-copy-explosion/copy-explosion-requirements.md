# 요구사항: copy-explosion (카피 폭발)

> **다음 단계 안내**: 이 문서는 PRD (기획 단계 요구사항만) 입니다. 다음 단계로 `tech-design` skill (또는 `/tech-design` 슬래시) 을 호출해서 `copy-explosion-tech-design.md` (기술 설계서) 를 만드세요. 기술 결정이나 구현 세부사항(파일 구조·프롬프트 포맷·린트 스크립트·병렬 실행 방식 등)은 여기 박지 마세요 — 그건 다음 두 산출물에 들어갑니다.

## 1. 배경/목적

`design-explosion-30-styles` 는 하나의 화면을 **30가지 디자인 스타일**로 폭발시킨다 — "어떻게 **보이나**" 축을 강제로 벌려 갤러리에서 비교하게 한다. 그 핵심 원칙은 **"콘텐츠는 고정, 스타일만 변수"** 다.

copy-explosion 은 그 **짝(twin)** 이다. 같은 도구를 **"뭐라고 말하나"(메시지 각도)** 축으로 뒤집는다. 랜딩페이지 히어로 섹션(헤드라인 + 서브카피 + CTA)을 하나의 제품에 대해 **30가지 서로 다른 메시지 각도**로 만들어 나란히 비교·조합하게 한다.

**왜 필요한가.** 랜딩 카피는 "봐야 아는" 문제다. AI에게 "카피 잘 써줘" 하면 매번 비슷한 톤 한두 개(대개 이득 소구 + 대담한 주장)만 나오고, "다른 느낌으로" 몇 번 더 시켜도 거기서 거기다. 어떤 각도가 이 제품에 맞는지는 머릿속으로 상상해서 비교해야 한다. 이 도구는 그걸 **30 각도로 강제로 벌려놓고 눈으로 비교**하게 만든다.

**이 프로젝트가 지키는 것 — 지어내지 않는다.** 30개는 *같은 하나의 제품*에 대한 30가지 표현 각도이지, 30개의 다른(허구의) 제품이 아니다. 입력에 없는 사실·수치·수상·고객사·기능을 만들어내면 그건 30갈래 거짓말이 된다. **제품 사실은 고정, 표현 각도만 변수** — design-explosion 의 "콘텐츠 고정, 스타일 변수" 를 메시지 축으로 옮긴 것이다.

**둘을 합치면.** copy-explosion 으로 **메시지**를 고르고, design-explosion 으로 **스타일**을 고르면 `스타일 × 메시지 = 랜딩페이지 완성`. 두 스킬은 별개가 아니라 조합되는 한 쌍이다.

## 2. 사용자 스토리 / 시나리오

*(간소 — 사용자층이 design-explosion 과 겹친다: 새 제품/기능의 랜딩·소개를 써야 하는 1인 개발자·창업자·기획자.)*

- **누가**: 새 제품/기능의 랜딩 히어로 카피 방향을 잡아야 하는 사람이
- **무엇을**: 제품을 한 줄로 던지면 30가지 메시지 각도의 히어로 시안을 갤러리로 받아
- **왜**: 어떤 메시지 각도가 이 제품에 맞는지 상상이 아니라 **눈으로 비교해** 고르고, 골라서 다음(스타일 입히기·실제 카피 확정)으로 넘어가기 위해.

## 3. 기능 요구사항 (FR)

### 받기 (입력)

- **FR-1**: 사용자는 제품/기능을 **한 줄 설명**으로 시작할 수 있다. (예: "DART 공시에서 숨은 관계를 찾는 GraphRAG")
- **FR-2**: 한 줄만으로 30개 각도를 제대로 뽑기에 정보가 부족하면, 스킬이 **최소한만 되물어** 채운다 — 되묻는 항목은 (a) 타겟(누구를 위한 것인가), (b) 핵심 이득 1개, (c) 뒷받침 근거/사실 1개. 사용자가 답을 모르거나 건너뛰면 그 각도군은 **약하게 표시**하되 진행을 막지 않는다.
- **FR-3**: 스킬은 되물어 얻은 것을 포함한 **제품 사실 스펙**(뭐하는 제품·타겟·이득·근거)을 2~3줄로 확정해 사용자에게 보여주고 진행한다. 이 스펙은 30개 전 시안에서 **한 글자도 다르지 않게** 고정된다 (FR-11 의 기준).

### 생성 (30 각도)

- **FR-4**: 확정된 제품 스펙 하나로 **30가지 메시지 각도**의 히어로 시안을 생성한다. 각 각도는 **이름이 붙어 있고**, 아래 6개 패밀리로 묶인다(정확한 30개 각도명은 tech-design 에서 확정):
  - **A. 감정 소구** — 공포/불안, 열망, 안도/평온, 자부심/지위, 소속감
  - **B. 논리·증거** — 데이터·수치, 비교·경쟁우위, ROI·비용, 보증·리스크제거, 메커니즘(왜 되는가)
  - **C. 사회적 증명** — 고객 목소리, 권위·전문가, 밴드왜건, 희소성·긴급
  - **D. 서사** — 스토리, 비포/애프터, 적(enemy) 프레이밍, 은유
  - **E. 인지 단계 (Schwartz)** — 문제 인식, 해결책 인식, 제품 인식, 가장 의식적(오퍼), 미인식(교육형)
  - **F. 구조·수사** — 질문형, 대담한 주장, How-to, 리스티클·숫자, 반전, 미니멀 한 문장
- **FR-5**: 각 각도는 그 각도의 **완성형 프롬프트**(무드·설득 구조·헤드라인 공식·금지·좋은/나쁜 예 포함)를 근거로 생성한다. 각도별 프롬프트는 서로 오염되지 않게 **각 시안은 자기 각도 프롬프트만** 참조한다.
- **FR-6**: 서브에이전트가 가능한 환경(Claude Code / Workflow 등)에서는 **각도당 1 에이전트로 병렬 생성**한다. 순수 채팅 환경이면 배치 직렬로 자동 폴백한다. 어느 쪽이든 30개가 모두 나온다.

### 산출물 (히어로 HTML)

- **FR-7**: 각 시안은 **실제 렌더링된 히어로 섹션 HTML** 이다 — 헤드라인 + 서브카피 + 1차 CTA(+선택적 보조 요소: 신뢰 배지·마이크로카피)가 배치까지 되어 화면으로 보인다.
- **FR-8**: 모든 시안은 **동일한 중립 스타일**(색·타이포·레이아웃 공통)을 쓴다. 스타일을 고정해야 **메시지만 변수**가 되어 각도끼리 정직하게 비교된다.
- **FR-9**: 각 파일은 **자기완결형 단일 `.html`** 이고, 하단 푸터에 `ANGLE {번호} · {각도명} · {패밀리}` 를 표기한다.
- **FR-10**: 30개 시안을 한눈에 보는 **갤러리 `gallery.html`** 를 자동 생성한다 — 패밀리별로 묶은 카드 그리드 + 각 카드 미리보기 + 새 탭 링크.

### 품질 규율 (린트 게이트)

- **FR-11 (진실성)**: 각 시안은 FR-3 의 제품 사실 스펙 안에서만 말한다. **입력에 없는 사실·수치·수상·고객사·기능을 지어내지 않는다.** 위반 시 그 시안을 표시하고 수정한다.
- **FR-12 (각도 구별성)**: 30개는 서로 **실제로 다른 각도**여야 한다. 두 시안이 사실상 같은 각도이면(헤드라인 접근이 겹치면) 표시하고 재생성한다.
- **FR-13 (히어로 제약)**: 각 시안은 히어로 형식을 지킨다 — 헤드라인·서브·CTA 각각의 길이 상한, **CTA는 1개**, 한 뷰포트 안에 완결. 구체 수치는 tech-design 에서 확정.
- **FR-14**: FR-11~13 을 **자동 게이트**(카피-린트)로 검사한다. 통과분만 사용자에게 제시하고, FAIL 은 해당 각도 프롬프트를 다시 읽고 수정 → 재검사한다.

### 비교·조합·확정

- **FR-15 (퍼널)**: 30개를 한 번에 비교하지 않는다. **패밀리 대표로 방향 좁히기 → 고른 패밀리 정독 → 결승 2~3개 나란히** 로 좁힌다.
- **FR-16 (조합)**: "3번 헤드라인 + 8번 CTA" 같은 축(헤드라인/서브/CTA/톤)별 병합 하이브리드를 만들 수 있다.
- **FR-17 (역추출 · 브릿지)**: 각도가 확정되면 그 각도를 **재사용 가능한 메시지 지침**(포지셔닝 한 줄 + 핵심 약속 + 톤 규칙)으로 역추출한다. 또한 확정 카피를 **design-explosion 으로 넘길 수 있게** 한다 — 그쪽에서 스타일을 골라 이 카피를 얹으면 랜딩이 완성된다(수동 핸드오프까지가 이번 범위, FR-연계는 §5 참조).

## 4. 비기능 요구사항 (NFR)

*(간소)*

- **NFR-1 (지어내지 않기 — 최상위 규율)**: FR-11 을 제품 전체의 1순위 규율로 둔다. 애매하면 만들지 말고 **약하게/보류**로 표시한다. 지어낸 이득 하나가 이 제품의 가장 큰 사고다.
- **NFR-2 (각도 커버리지)**: 최종 제시분은 **6개 패밀리를 모두** 포함한다(한 패밀리에 쏠리지 않는다). 30개 중 린트 통과분이 30 미만이어도 각 패밀리 최소 1개는 남긴다.
- **NFR-3 (자기완결)**: 각 시안은 외부 런타임 의존 없이 단일 HTML 로 열린다(한글/디스플레이 폰트 CDN 만 허용). JS 는 연출 전용 — 꺼져도 카피는 온전히 읽힌다.
- **NFR-4 (언어)**: 출력 카피는 **입력 제품의 언어**를 따른다(기본 한국어). 실존 인물·기업명은 예시로도 쓰지 않는다.
- **NFR-5 (병렬 안전)**: 병렬 생성 시 각 에이전트는 **공유된 제품 스펙(고정)** 만 공유하고 자기 각도 프롬프트만 읽는다 — 각도 간 카피가 새지 않아야 한다.

## 5. 범위 밖 (Out of Scope)

- **실제 A/B 테스트 · 전환율 측정** — 어떤 각도가 실제로 전환되는지 검증은 사용자 몫. 이 도구는 "비교할 30개를 만든다" 까지.
- **각도 × 스타일 자동 결합** — design-explosion 과의 연계는 **수동 핸드오프**(확정 카피를 넘기는 것)까지. 30 각도 × 30 스타일 자동 매트릭스 생성은 다음 버전.
- **롱폼 카피** — 히어로(헤드라인/서브/CTA)만. 본문·이메일 시퀀스·광고 카피 세트는 범위 밖.
- **이미지 · 비주얼 에셋 생성** — 히어로는 텍스트 + 레이아웃까지. 실제 사진·일러스트·로고는 만들지 않는다.
- **다국어 동시 생성** — 한 번에 한 언어.
- **브랜드 보이스 학습 · 영구 저장 · 계정** — 세션 단위로 끝난다.

## 6. 수용 기준 (Acceptance Criteria)

- **AC-1**: 제품 한 줄 설명을 주면, 정보가 부족할 때 스킬이 (타겟/이득/근거) 중 부족한 것만 되묻고, 확정된 제품 스펙 2~3줄을 보여준 뒤 생성으로 넘어간다.
- **AC-2**: 30개(또는 린트 통과분) 히어로 시안이 생성되고, 각각 헤드라인·서브·CTA·`ANGLE n·각도명·패밀리` 푸터를 갖는다.
- **AC-3**: 30개는 **모두 같은 중립 스타일**로 렌더된다(색/타이포/레이아웃 공통) — 눈에 보이는 차이는 카피뿐이다.
- **AC-4**: 제시된 시안 집합이 **6개 패밀리를 모두** 포함한다.
- **AC-5 (진실성)**: 입력 스펙에 없는 수치·수상·고객사를 넣은 시안은 린트에서 잡혀 수정/제외된다. (예: 입력에 "고객 1만 명" 이 없는데 헤드라인에 "1만 고객" 이 있으면 FAIL)
- **AC-6 (구별성)**: 최종 제시분에 사실상 같은 각도의 중복 시안이 없다.
- **AC-7 (히어로 제약)**: 모든 시안이 CTA 1개, 헤드라인/서브 길이 상한 이내, 한 뷰포트 완결을 지킨다.
- **AC-8**: `gallery.html` 을 열면 30개가 패밀리별로 묶여 한눈에 보이고, 각 시안 새 탭 링크가 동작한다.
- **AC-9**: 서브에이전트 가능 환경에서는 병렬로 생성되고, 순수 채팅 환경에서는 직렬로 폴백해도 30개가 모두 나온다.
- **AC-10**: 각도 하나를 확정하면, 재사용 가능한 메시지 지침(포지셔닝 + 핵심 약속 + 톤)이 역추출되어 나온다.
