## 화면 2 — 예약 폼 `screen-2-booking.html` · 원형 **E 폼·입력형**
**필수 region**: `form` · 권장 `steps` `summary` `identity` `action`
- 상단: 같은 로고 + 브레드크럼 `검색 결과 › 남산 그랜드 호텔 › 예약`
- 단계: `① 객실 ─ ② 투숙객 ─ ③ 결제` 현재 **② 투숙객 (2/3)**
- 폼 필드(상태 5종 시연):
  - 대표 투숙객 이름 `정하람` (완료 ✓)
  - 이메일 `haram@` — **에러: 이메일 형식을 확인해 주세요**
  - 연락처 `010-2481-0000` (포커스)
  - 요청사항 (textarea) `늦은 체크인 예정입니다. 22시 이후 도착할 것 같은데 가능할까요? 가능하면 고층 금연실로 부탁드립니다.` ← 긴 입력
  - 조식 추가 `2인 (+₩36,000)` (선택)
  - 결제 수단 (비활성 — 3단계에서 입력)
  - 약관 동의 체크박스 (미체크, 필수)
- 에러 요약 배너: `필수 항목 1개를 확인해 주세요.`
- 요약 패널 `summary`: **남산 그랜드 호텔** · `9/12–9/14 (2박)` · `성인 2` · 객실 `디럭스 더블` · 소계 `₩364,000` + 조식 `₩36,000` = **₩400,000**
- 액션: 주요 `결제로 이동` + 보조 `객실 다시 고르기`

## 세트 규칙 (3화면 공통)
- **스타일은 03 트렌디 SaaS 하나로 고정** — 색·타이포·간격·시그니처(카운트업·숨쉬는 도트)가 세 화면에서 동일해야 한다. 흔들리면 세트가 아니다.
- 상단 로고·사용자 이니셜·브랜드 톤을 3화면에서 똑같이.
- 호텔·인물 전부 가상. 숫자·문구 한 글자도 안 바꿈.
- 각 화면은 자기 원형의 `data-region` 필수 영역을 갖고, **다른 원형의 금지 영역을 넣지 않는다** (예: 예약 폼에 KPI 금지).
