> ## 막연한 "예쁘게 해줘"를, 30가지 확실히 다른 방향으로
>
> 도메인 하나만 던지면 **같은 화면을 30가지 디자인 스타일로 폭발**시킵니다.
> 콘텐츠는 고정하고 **시각 언어만 30갈래**로 벌려서, 갤러리에서 나란히 보고 방향을 고릅니다.
>
> <sub><i>Turns a vague "make it nice" into 30 concrete design directions — same content, 30 visual languages, side by side.</i></sub>

---

## 두 개의 축 — 스타일 × 메시지

이 플러그인에는 **스킬 두 개**가 들어 있습니다. 랜딩 하나를 만들려면 결정이 둘이거든요.

| 스킬 | 축 | 무엇을 30갈래로 | 언제 |
|---|---|---|---|
| **design-style-explorer** | 어떻게 **보이나** | 30가지 디자인 스타일 | 비주얼 방향이 안 정해질 때 |
| **copy-angle-explorer** | 뭐라고 **말하나** | 30가지 메시지 각도 (6패밀리) | "뭐라고 소개할지" 막힐 때 |

> **스타일 × 메시지 = 랜딩 완성.** 카피를 먼저 고르고 스타일을 입히거나, 반대로 해도 됩니다.

두 스킬 모두 같은 규율을 씁니다 — **한 변수만 바꾼다.** 스타일 탐색에선 콘텐츠를 고정하고, 카피 탐색에선 제품 사실을 고정합니다. 그래야 비교가 정직합니다.

그리고 각각 **자동 게이트**가 붙어 있습니다:

- `style-lint` — 그 스타일의 금지를 어겼나
- `archetype-lint` — **이 도메인의 화면이 맞나** (대화형 제품에 KPI·도넛이 박히는 사고를 잡습니다)
- `copy-lint` — 제품 사실에 없는 수치·최상급을 지어냈나

## 이런 적 있죠?

새 화면 디자인 방향을 잡아야 하는데 —

- AI에게 "예쁘게 디자인해줘" 하면 **매번 비슷한 톤** 한두 개만 나오고,
- "다른 느낌으로" 몇 번 더 시켜도 결국 **거기서 거기**,
- 좋은 방향인지 아닌지는 **머릿속으로 상상**해서 비교해야 하고,
- 막상 뽑아보면 큰 숫자 · 긴 제목 들어가는 순간 **레이아웃이 무너지고** …

이 도구는 그걸 **30갈래로 강제로 벌려놓고 눈으로 비교**하게 만듭니다.

## 뭐가 다른가요?

| | 그냥 AI에게 "디자인 해줘" | design-explosion |
|---|---|---|
| **결과** | 매번 비슷한 톤 1~2개 | 확실히 다른 **30가지** |
| **비교** | 머릿속으로 상상 | **갤러리에서 나란히** |
| **일관성** | 스타일마다 될 때도 안 될 때도 | **린트 게이트**로 매번 규칙 준수 |
| **콘텐츠** | 예쁜 더미데이터만 | 큰 숫자 · 긴 제목 · 빈 상태 **스트레스 테스트** |
| **다음 단계** | 마음에 안 들면 처음부터 | 방향 골라 **세트 생성 → 디자인 시스템** |

## 3단계로 시작

**1. 설치** — Claude Code 안에서:

```
/plugin marketplace add deokjinlog/design-explosion-30-styles
/plugin install design-explosion-30-styles@design-explosion-30-styles
```

**2. 도메인 던지기** — 세 가지 방법이 있어요. 아래로 갈수록 정확합니다:

```
이커머스 셀러 대시보드 디자인 30개 스타일로 뽑아줘        ← 도메인만
연서, 대화형+읽기형 원형으로 30 스타일 뽑아줘              ← 원형 직접 지정
/home/me/myproject 기준으로 디자인 시안 뽑아줘            ← ⭐ 프로젝트 폴더
```

**프로젝트 폴더를 주는 게 제일 좋아요.** 요구사항·코드를 읽어서 화면을 **역산**하거든요. 실제로 `FR-1~19` 가 있는 프로젝트를 넣었더니 질문이 거의 없이 정확한 화면이 나왔습니다.

그리고 어느 방법이든 **먼저 물어봅니다** — 제일 중요한 건 이 한 문항이에요:

> **"이 화면에서 사용자가 하는 단 하나의 행동은?"**
> (훑는다 / 대화한다 / 읽는다 / 고른다 / 입력한다 / 흘려본다 / 만든다 / 찾는다 / 처음 배운다 / 로그인한다 / 내 정보를 본다)

이 답이 **화면 원형**을 정하고, 원형이 화면 구성을 정합니다.

**3. 갤러리에서 고르기** — 나란히 훑고 마음에 드는 번호를 고르면 끝.

<sub>스크린샷이 아니라 살아있는 화면이에요. 시안마다 따로 렌더해서 서로 CSS가 안 섞이고, 실제 데스크톱 크기 그대로 비교됩니다.</sub>

> *"디자인 시안 뽑아줘"* · *"30개 스타일로 보여줘"* · *"○○ 도메인 디자인 보여줘"* — 뜻만 통하면 발동해요. 100% 확실히 하려면 `/design-style-explorer` 로 직접 부르면 됩니다.

<sub>산출물은 <code>docs/design-gallery/&lt;날짜&gt;-&lt;도메인&gt;/</code> 에 <code>.html</code> 시안 + 비교 갤러리로 쌓여요.</sub>

## 도메인에 맞는 화면부터 — 원형 11종

**스타일이 맞아도 화면이 틀릴 수 있어요.** 실제로 겪은 사고입니다:

문학 스토리 서비스(대화로 이야기를 제안받는 제품)에 시안 30개를 뽑았는데, 전부 `들은 스토리 1,284,000편` · `목표 달성률 65%` · `축 구성비 도넛` 이 박혀 있었습니다. 대시보드 구성을 그대로 이식한 거예요. **린트는 30/30 통과**였습니다 — 스타일 규칙은 다 지켰으니까요. 근데 **비교할 수 없는 갤러리**였습니다. 그 제품의 화면이 아니었거든요.

그래서 **화면 원형**을 먼저 고릅니다:

| | 원형 | 무엇이 주인공인가 |
|---|---|---|
| A | 대시보드형 | KPI · 차트 · 표 |
| B | 대화형 | **대화 흐름** (곁다리 패널 금지) |
| C | 읽기형 | **긴 본문** — 승부처는 타이포그래피 |
| D | 컬렉션형 | 카드 그리드 · 필터 |
| E | 폼·입력형 | 입력 필드 · 단계 |
| F | 피드형 | 시간순 항목 |
| G | 에디터·캔버스형 | 편집 영역 · 툴바 |
| H | 검색·결과형 | 검색어 하이라이트 |
| I | 온보딩형 | 슬라이드 · 진행 표시 |
| J | 인증형 | 단일 카드의 무게 |
| K | 프로필·설정형 | 행 밀도 · 토글 |

원형이 바뀌면 **비교하는 대상 자체가 달라집니다.** 대시보드는 숫자 타이포·차트 색으로 갈리고, 읽기형은 본문 측정폭·행간·인용 처리로 갈려요. 읽기형에 차트를 넣으면 그 비교가 망가집니다.

<sub>원형 목록은 지어낸 게 아니라 <a href="https://github.com/luileito/enrico">Enrico</a>(MIT · RICO 72k 모바일 UI에서 추린 1,460 UI를 사람이 20개 토픽으로 분류)로 검증했어요. 대조하니 <code>Tutorial</code>(3위) · <code>Login</code>(2위) · <code>Profile+Settings</code> 가 빠져 있어서 I·J·K로 추가했고, 반대로 Enrico엔 없는 <b>대시보드</b>(모바일 데이터셋이라 0개)는 웹·SaaS용으로 채웠습니다.</sub>

## 게이트가 둘이에요

두 검사는 **직교**합니다. 위 사고에서 ①은 만점을 줬고 ②가 잡았어요.

| 게이트 | 무엇을 보나 | 사고본 결과 |
|---|---|---|
| ① `style-lint` | 그 **스타일**의 금지를 어겼나 | 30/30 PASS ✅ |
| ② `archetype-lint` | 이 **도메인의 화면**이 맞나 | **119건 위반** ❌ |

②가 FAIL 나면 **시안을 고치지 않습니다.** "1단계 콘텐츠 스펙이 틀렸다"고 알려줘요 — 시안만 고치면 다음 배치에서 그대로 재발하니까요.

정확도를 위해 생성물은 의미 영역에 속성을 답니다. **의미는 고정, 레이아웃은 자유**예요 — 벤토는 여전히 비대칭으로, 스위스는 그리드로 배치합니다:

```html
<section data-region="conversation"> … </section>
<article data-region="story-body">   … </article>
```

```bash
python3 scripts/archetype-lint.py docs/design-gallery/<날짜>-<도메인>/
```

<sub>폴더에 <code>_spec.json</code>({"archetype": "B", …})을 두면 원형을 자동으로 읽어요. 속성이 없는 예전 파일은 키워드 추측으로 폴백하되 "부정확" 이라고 알려줍니다.</sub>

## 왜 30개가 다 다른가 (무작위가 아니에요)

AI는 그냥 두면 자기가 편한 톤으로 수렴해요. 이 도구는 스타일마다 **각본**을 박아둡니다.

- **스타일당 전용 프롬프트** — 매력 축 · 팔레트(hex) · 타이포 · 시그니처 · 금지 목록 · 크리틱 루프까지. 무작위가 아니라 각각 튜닝된 지시서예요.
- **린트 게이트 2종** — "브루탈리즘인데 그라디언트?"(스타일)와 "대화형인데 KPI?"(원형)를 코드가 자동으로 잡아서 FAIL → 수정 → 다시 그려요. 30개를 사람이 일일이 눈으로 검수하지 않아도 됩니다. ([위 "게이트가 둘이에요"](#게이트가-둘이에요))
- **실무 마감** — 8px 그리드 · hover/focus 상태 · CSS 변수. Linear · Stripe 볼 때 그 촘촘한 느낌.
- **최악 콘텐츠 테스트** — 긴 제목 · 큰 숫자 · 결측 · 빈 상태를 넣어도 안 무너지게. 템플릿은 베스트 케이스를 팔지만, 실제로 쓸 땐 최악 콘텐츠가 들어오니까요.

<sub>린트는 "이 스타일이 맞나"(3초 무드)를, 최악 콘텐츠 테스트는 "오래 써도 안 무너지나"(견고성)를 봅니다. 둘은 직교하는 두 축이라, 둘 다 통과해야 진짜 쓸 만한 시안이에요.</sub>

## 이렇게 흘러가요

```
도메인(또는 프로젝트 폴더)
  → 원형 결정 (역산 · 인터뷰)  →  콘텐츠 스펙 고정 + _spec.json
  → 30 스타일 병렬 생성  →  게이트 ① style-lint  +  게이트 ② archetype-lint
  → 갤러리에서 방향 결정 (퍼널)  →  나머지 화면 일관 생성
  → 디자인 시스템(토큰 · 규칙) 역추출
```

먼저 **한 화면을 30가지로** 뽑아 방향을 정하고, 정해지면 그 스타일로 나머지 화면을 이어가요.
마지막엔 색 · 간격 · 컴포넌트 규칙을 **디자인 시스템으로 역추출**해서, 이후 화면은 그 규칙대로 찍어냅니다.

**비용을 아끼려면 퍼널을 앞으로 당기세요.** 30개를 다 도메인 콘텐츠로 뽑는 건 낭비예요 — 결정이 두 종류라서:

| 결정 | 내 콘텐츠가 필요한가 |
|---|---|
| *"브루탈리즘이 이 서비스에 맞나?"* | ❌ 시각 언어만 봐도 3초에 잘림 |
| *"이 스타일이 600자 본문을 견디나?"* | ⭕ 반드시 필요 |

그래서 **1라운드는 아래 30 스타일 갤러리(이미 배포됨)로 3초컷 → 6~8개만 도메인 생성 → 결승 2~3개** 가 효율적입니다. 실측으로 토큰이 1/4로 줄었어요.
